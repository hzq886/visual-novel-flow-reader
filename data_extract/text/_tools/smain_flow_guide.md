# SMAIN フローバイトコード 解析ガイダンス

対象: `md_scr.med`（MED エンジン）の `SMAIN` エントリ ＝ ゲーム全体の進行を司るマスタースクリプト。
全ルート・全分岐・全エンドのシーン遷移をバイトコードで保持する。`extract-flow.py`（HU-15）の出典。

> 本書は「本プラン最大の不確実性（未知バイトコード解析）」の解読結果を記録する。
> 復号（鍵 `tauromin` / `_VIEW` クリブ法）までは [text_unpack_guide.md](text_unpack_guide.md) を参照。

## 0. サマリ

| 層 | 結論 |
|---|---|
| `SMAIN` 復号 | 他エントリと同じ。16B サブヘッダ平文＋ payload を鍵 `tauromin` で加算復号 |
| サブヘッダ | `u32 a / u32 unk1 / u16 c1 / u16 c2 / u32 d`。**`unk1`=レコード列終端オフセット, `c2`=ラベル表エントリ数, `d`=レコード数** |
| payload | 3 層構造（レコード列 → ラベルオフセット表 → 文字列表） |
| 命令 | `u16 連番 + u8 長さL + L バイトのデータ`。**データ内側 先頭バイトが実オペコード** |

本作 `SMAIN`: payload 11547B、レコード 1230、文字列 326、ラベル表 25。

---

## 1. payload の 3 層

```
offset
0x0000  ── レコード列 ──            u16 連番(0,1,2…) + u8 長さL + データ[L]
unk1    ── ラベルオフセット表 ──    u16 × c2（SMAIN_* hub の正規バイトオフセット）
        （＋末尾に SJIS のタイトル文字列断片）
strOff  ── 文字列表 ──             NUL 区切り（シーン名・hub ラベル名）
```

- **レコード列の終端検出**: 連番 u16 が 0 から 1 ずつ増える制約で読み、破綻した位置が終端。
  サブヘッダ `unk1` がこの終端を指す（= ラベル表開始）。レコード数はサブヘッダ `d` と一致。
- **文字列表の起点検出**: payload 中で最初に正規表現 `\d{3}_[A-Z]`（シーンコード）が現れる位置。
- 文字列は 1 始まりの索引で参照される（索引 1 = 先頭文字列）。

## 2. 命令フォーマット

```
record = u16 lineno ; u8 len ; u8 data[len]
```

- `len`（＝外側オペコード）は**そのままデータ長**。`lineno` は 0 起算の通し番号（再パック用）。
- 意味は **データ先頭バイト（内側オペコード）** で決まる。本作で観測した内側オペコード:

| 内側 op | 形 | 件数 | 意味 |
|---|---|---:|---|
| `0x1b` | `1b <u16 idx>` | 363 | **シーン呼び出し**（idx = 文字列表）。`STAFF_ROLL` も含む |
| `0x1c` | `1c <u16 idx>` | 50 | **hub ラベルへの（条件）goto**（idx = 文字列表）。`SMAIN_*` / `NORMAL_END` / `TRUE_END` |
| `0x08` | `08 01 <u16 off>` | 65 | 局所**前方**スキップ（提示イディオム。物語分岐ではない） |
| `0x09` | `09 01 <u16 off>` | 65 | 局所**後方**ジャンプ（直前シーンへの back-pointer。65/65 が手前を指す） |
| `0x1d` | `1d 00 <id> f5 f5 eb 01 ff`（len8） | 5 | **select/フラグ判定マーカー**（id = ローカル選択分岐 id） |
| `0x02` `0x03` `0x06` 等 | — | — | 演出・区切り・プロローグ初期化（フロー復元では無視） |
| `0x07` | `07 <k>`（len2） | 25 | **ラベル定義マーカー**（直後のレコードが label[k]。k=0..24 昇順。HU-22） |

`0x08`/`0x09` の `off` と ラベル表は **バイトオフセット**（レコード境界に一致）。`0x1c` の goto 先解決には
ラベル表（hub 名 → 正規オフセット）を使う。**合流（同名 hub の複数 `0x1c`）は名前一致**で表現するが、
**hub の継続先（合流後 goto がどのブロックへ行くか）はラベル表で解決する**（HU-22。§3.10）。各ラベルは
直前に `0x07 <k>` マーカーを持ち（25 個＝ラベル数と一致）、レコード列を 25 ブロックに分割する。

## 3. フロー復元（extract-flow.py の写像）

1. レコード列を走査し、`0x1b`(scene) / `0x1c`(hub|end) / `0x1d`(select) を**順に**イベント化。
2. ノード = ユニーク名（シーンコード / hub 名 / END 名）＋合成 `start`。
   エッジ = 隣接イベント連接（再入シーンは同一ノードに集約＝複数 in/out）。
3. **線形チェーン収縮**: in/out 次数 1 の直列 arc ランを 1 ノード（`scenes[]` 保持）へ圧縮。
   hub / end / start／**ルート分岐 source・case-head**（§3.8）は境界として残す → 読みやすい完全
   グラフ（本作 76 ノード / 100 エッジ。HU-21 で len-8 switch を真の分岐へ分離した結果）。
4. `0x1d` の直後イベントへ `condition.flags = ["SEL_xx"]` を付与（収縮後 hub 境界に残るもののみ）。
5. 層化レイアウト（前方エッジ最長距離で列・キャラレーンで行）→ `pos` 付与。

検証: 線形 `0x1b` 列はシーン表順＝攻略のシーン順を再現（`build_ayan_end1.py` の SCENES は
このうち綾菜 END1 経路を人手で辿った一例）。`npm run validate` で flow↔原テキスト相互照合。

## 3.6 選択肢メニュー文言の i18n 抽出（HU-18）

選択肢文言は SMAIN ではなく**各シーン脚本側**にある。シーン脚本 bytecode は SMAIN と別形式で
均一に解けないが、抽出済みテキスト（`md_scr_text_{jp,cn}`）に**権威ある印**がある:

- **選択肢オプション = 直後に `[id] <SCENECODE>_NN_MM` が続く本文行**（`NN`=メニュー番号, `MM`=選択肢番号）。
  例: `[text] 綾姉は特別な存在だ` → `[id] 002_AYAN008A_01_01`。
- この選択肢 ID（シーンコード＋`_NN_MM`）は**ボイス ID（`CHAR_SCENE_serial` 形式）と明確に別形式**。
  誤検出回避のため **同一 `NN` に 2 択以上ある**メニューのみ採用。
- cn 版は `[ascii] <SCENECODE>_NN_MM` ＋ `[cn]` 本文で同様に取れる。

`extract-flow.py` の `extract_choices()` がこれを実装し、各メニューを所有ノードの `choices`
（`{scene, options:[{jp,cn}]}`、[ADR 0002](../../docs/adr/0002-data-schema.md)）に付与する。
本作は **19 メニュー / 38 選択肢**（ID 方式 18 ＋ `_VIEW` 方式 1）を jp/cn 両言語で抽出
（FlowMap がノードに「◇A／B」表示）。

注意点:
- `_VIEW` はほぼ全シーンの末尾マーカー（選択肢マーカーではない）。素朴な `_VIEW`/`BG_BLACK`
  アンカーはシーンタイトル `\N`・背景ノート `#…`・ナレーション末尾を誤検出する。
- **`_VIEW` 方式メニュー（選択肢 ID を持たないもの。HU-19・実装済）**: `006_TUBA001B`（「後悔して、
  すぐに止める」/「背徳に溺れて、このまま続ける」）等。`BG_BLACK`/`_VIEW` 直前から後方走査し、
  **ナレーション末尾（約物）/ `\N` / `#` / 非 CJK を境界・除外**して 2 択以上だけ採る
  （`_view_menu()`）。全 319 シーンで誤検出 0、本作の該当は 1 シーン。

## 3.7 `_DEF` フラグ解決（HU-20・Part A 実装済）

SMAIN のフラグ操作（`0x06` init: `06 00 <slot> f5 eb 00 ff` / `0x1d` select:
`1d 00 <slot> f5 f5 eb 01 ff`）は **フラグスロット番号**を直接参照する。`_DEF.txt` がスロットを
実フラグ名へ対応づける**シンボル表**:

- 変数別名: `[id] S521\tS60` ＝ スロット `S60` の名前は `S521`。
- 手動フラグ: `[text] S50手動1\tS55`。
- **ルート軸フラグ**（`[note]`、空白区切り）: `#軸1_1 S70` / `#軸2_1 S71` `#軸2_2 S72` /
  `#軸3_1`〜`#軸3_3`（S73-75）/ `#軸4_1`〜`#軸4_4`（S76-79）/ `#軸5_1`〜`#軸5_5`（S80-84）。
  **軸1〜5 = ルート決定軸**（分岐数 1/2/3/4/5）。

`extract-flow.py` の `def_flag_table()`/`resolve_flag()` が select マーカーのスロットを解決し、
`FlowEdge.condition.flags` に実フラグ名で載せる（例 `SEL_47` → `S71/軸2_1`）。SMAIN の select は
スロット 61,62,63（`S522`〜`S524`・TUBA 系）/ 71,72（`軸2_1/2`・MAKO 系）を読む。

**HU-21 で完了**: 各**選択肢オプション → 分岐先ノード**の厳密紐付け（§3.8）。シーン脚本 bytecode の
フラグ set opcode を RE し、SMAIN の len-8 switch と突き合わせて解決した。

## 3.8 選択肢オプション → 分岐先ノードの厳密紐付け（HU-21・実装済）

### シーン脚本 bytecode の解読

各シーン脚本（`002_AYAN008A` 等のエントリ）も SMAIN と**同じ命令形式** `u16 lineno + u8 len +
データ[len]`。ただし `lineno` は**ソース行番号で非減少**（同 lineno に複数命令あり）＝ SMAIN の
「連番 +1 厳密増加」自己同期が効かない。**len 前置で線形に読めば解ける**（過去 intractable と
された主因はこの誤同期）。3 層構造（records → label 表 `c2` 個 u16・**unk1 から。rec_end の 1B 後** →
文字列表）も SMAIN と同形。subheader `d` = 最大 lineno。

| 内側 op | 形 | 意味 |
|---|---|---|
| `0x06` | `06 00 <slot> f5 eb <val> ff`（len7, byte1=00=通常変数） | **フラグ代入** `S<slot> := <val>` |
| `0x06` | `06 00 <slot> f5 00 <slot> ee eb 01 ff`（len10） | 加算カウンタ（`S11 += 1` 等。選択回数） |
| `0x1c` | `1c <u16 idx>` | 選択肢オプション（idx = 表示文字列） |
| `0x0a` | `0a <u16 idx> 00 00 f5 00 00 ff` | メニュー文/選択肢 ID（`<scene>_NN_MM`）/ナレーション |

**メニュー文法**: 1 オプション = `[06 代入…] 1c <選択肢文> 03 0a <選択肢ID>` を `08`(前方skip)
/`09`(back-ptr) の提示ループで囲む。代入と選択肢文は近接（表示順≠値順あり。例 AYAN008A は局所
S12/S13 を 0/1 で書く非分岐の演出選択肢）。

### SMAIN len-8 switch の case 解析

ルート分岐は SMAIN の `1d 00 <slot> f5 f5 eb 01 ff`（len8 = switch）。直後に `08 01 <u16 off>`
（off は **byte オフセット**＝データ[2:4]）でガードされた **case-block** が並び、`1e` で区切られる。
**case の順序＝フラグ値（1 始まり）**。block 内の `1b`=scene / `1c`=hub-goto が分岐先。
本作の 5 switch: `S61/62/63`（TUBA 系=`S522-524`）・`S71/S72`（軸2）。
> 検証: `S71==2` の len-7 等値テスト（`1d 00 47 f5 eb 02 ff`）が MAKO017 系へ向かい、len-8 switch の
> case2（値2＝「より深い」選択肢が書く値）と独立に一致。**case 順＝値**を裏取り。

### 紐付けと flow 表現

`scene_choice_flags()` が各シーンの `06` 代入を選択肢文へ紐付け（ID 式は近接 `1c`/`0a`、`_VIEW` 式は
順序）、`smain_switches()` が switch を value 順 case へ分解する。`apply_switches()` が **source
（フラグを書いたメニューシーン）→ 各 case-head** に **条件（フラグ=値）＋選択肢ラベル**のエッジを
張り、case 間のレイアウト隣接由来の**誤った線形エッジ**（`case_v 末尾 → case_{v+1} 先頭`、例
`MIX03→MAKO001B`）を除去する。`FlowChoice.options[*]` には `flag`（例 `S71/軸2_1=2`）/`target`
（分岐先ノード id）/`targetTitle` を付与（[ADR 0002](../../docs/adr/0002-data-schema.md)）。

注意・限界:
- シーン内ラベル `MIX01`/`MIX02` は画像トランジションのローカル印で分岐先ではない（誤用注意）。
- hub の合流後 goto（`SMAIN_MIX*` の継続先）は **HU-22 で解決済**（§3.10）。`SMAIN_MIX01`/`MIX03` の
  sink は解消。len-7 等値テストの「任意挿入シーン」の choice 紐付けは **HU-23 で解決済**（§3.8.1）。

### 3.8.1 len-7 等値テストの任意挿入シーン紐付け（HU-23・実装済）

主要ルート分岐（§3.8 の len-8 switch・5 個）とは別に、SMAIN には **len-7 等値テスト**
`1d 00 <slot> f5 eb <val> ff`（＝ `if S<slot>==<val>`）が 27 個ある。構造:

```
1d 00 <slot> f5 eb <testval> ff   if S<slot> == <testval>
08 01 <u16 off_true>              ガード: 偽なら off_true へ前方 skip
  … TRUE ブロック（1b=scene / 1c=hub-goto）…
09 …                             back-ptr（無視）
1e                               区切り（＝ELSE 開始。無い場合あり）
08 01 <u16 off_false>            ガード
  … ELSE ブロック …
09 …
```

これらは len-8 switch（END へ抜ける**恒久分岐**）と違い、TRUE/ELSE ブロックの再生後に**後で合流する＝任意挿入**。
トリガ選択肢を選ぶと挿入ブロックが再生され、選ばなければ素通り（or 別ブロック）して合流する。

**位置認識の紐付け**: 同一スロットが脚本内で scratch 的に**再利用される**（`S12` は AYAN008A/AYAN010A/SUZU004E、
`S16` は楓サブアーク全域）。よって `slot→value` の大域表では曖昧（同値が別位置で別ブロックを指す）になる。
各**書き込みシーン**（`06 00 <slot> f5 eb <val> ff` を持つ選択肢シーン）の SMAIN イベント位置の**直後で最も近い**
同スロット len-7 テストを採り、`val==testval` なら TRUE、それ以外は ELSE ブロックの**先頭 dest** を挿入先とする。

**flow 表現**（`smain_len7_tests()` / `scene_choice_inserts()` / `to_flow()`）: 各選択肢オプションに `flag` を付け、
ブロック先頭 dest の種別で出力先を分ける——
- 先頭が **scene**（`1b`）→ 任意挿入 `options[*].inserts`（＋`insertsTitle`）。再生後に合流する。
- 先頭が **hub-goto**（`1c`、別ルートへの恒久分岐＝`S12`→AYAN01/SUZU01 等）→ `target`（len-8 と同義）。
- ブロックが空（素通り）→ `flag` のみ。
- len-8 switch スロットは既存 `target` を優先（`S71` は len-8 分岐＋len-7 の MAKO017 挿入を兼ねるが二重付与は避ける）。

挿入ブロックの **CFG 連結**自体は HU-22（§3.10 手順 4）で済んでおり、本節は**選択肢 → 挿入ブロックのメタ紐付け**を担う
（再生エンジン nav の挿入再生＝合流復帰は別途・本節の対象外）。出力例: SUBTM001A「真琴にも手を出そう」(S75=1)
→ `inserts`=005_MAKO003A、MAKO005E「無視する」(S74=1) → `inserts`=005_MAKO006A、AYAN008A「綾姉は特別な存在だ」(S12=1)
→ `target`=SMAIN_AYAN01。

## 3.10 hub 合流後 goto の継続先解決＝full CFG 化（HU-22・実装済）

`0x1c <hub>` goto の合流は名前一致で表すが（§2）、**合流後にどのブロックへ継続するか**は HU-21 まで
未解決で、`SMAIN_MIX01`/`MIX03` が出力エッジ無しの sink だった。HU-22 で **ラベル表で継続先を解決**した。

### ラベル表 ↔ hub 名の対応

- サブヘッダ `unk1`（= `rec_end + 1`）から `c2`(=25) 個の u16 ＝ **昇順バイトオフセット**。各々レコード境界に
  一致し、レコード列を 25 ブロック（＋先頭の ENTRY ブロック）に分割する。各ラベル直前に `0x07 <k>`
  （k=0..24）マーカーがある（25 個＝ラベル数。裏取り）。
- **対応規則は文字列表順の単純 zip では決まらない**（`NORMAL_END`/`TRUE_END` 等の終端に body が誤対応）。
  **制御フロー**で確定する: 大分岐（`0x2a` ジャンプテーブル＋`0x0a` 条件、slot14=翼踏込/slot15=真琴）が
  `SISTER01/TUBA01/TUBA02/MAKO01/TUBAMAKO01/TUBAMAKO02` へ分岐する点を起点に、各ブロックの goto・
  fall-through・シーンコード連続性を辿って命名し、route-map（二次ソース）の各ルートと突き合わせ検証した。
- 確定マップ（hub→label index、22 hub↔ブロック L0–L21、L22/L23=STAFF_ROLL slot10=1/2・L24=cleanup は端末）:
  `TUBA03`=L0, `MIX01`=L1, `MIX02`=L2, `MIX03`=L3, `MIX04`=L4, `SISTER01`=L5, `SISTER02`=L6, `AYAN01`=L7,
  `AYAN02`=L8, `AYAN03`=L9, `SUZU01`=L10, `SISTER03`=L11, `SISTER04`=L12, `TUBA01`=L13, `TUBA02`=L14（TRUE_END 到達）,
  `MAKO01`=L15, `TUBAMAKO01`=L16, `TUBAMAKO06`=L17, `TUBAMAKO02`=L18, `TUBAMAKO03`=L19, `TUBAMAKO04`=L20, `TUBAMAKO05`=L21。

### flow 表現（`resolve_hub_continuations`）

1. **goto-run ファンアウト**: 連続する `0x1c`（大分岐 dispatch・len-7 select 列）は直前 scene から並列分岐
   （隣接では先頭 1 本しか張られない）。
2. **hub 継続先**: 各 hub の近似 out-edge（レコード隣接由来）を全消去し、ラベル表で解決したブロック先頭
   イベントへの単一エッジに置換。
3. **fall-through 経由**: scene 終端ブロックが hub ブロックへ落ちる所は scene→hub に張り替え、合流点を経由
   （例 `MIX01` ブロック `TUBA001D` → `MIX02` → `AYAN002A`）。
4. **挿入シーンの連結**: len-7 等値テストの else ケース（`SUBTM` 等）は governing select の直前シーンへ接続し
   CFG を連結（choice/flag 紐付けは HU-23）。

検証: 合流 hub の sink 0、全ノード start から到達かつ各ノードから END 到達可能（双方向連結）、JP/CN 構造同一。
出典: `extract-flow.py`（`SMAIN_HUB_LABEL` / `resolve_hub_continuations`）。

## 3.9 効果音 se / BGM の RE（HU-28）

音声チャンネルは `_SYSTEM` が定義する: `VOL_BGM:1` / `VOL_CV:2`（台詞）/ `VOL_SE:3` /
`VOL_LPSE:3`（ループ se）/ `VOL_BGCV:4`（`BGV_*` 喘ぎ）/ `VOL_SYS:5`。

### se（効果音）= bytecode に存在・抽出可

- シーン脚本の **`0x6c`（`6c <u16 idx>`）が se 再生命令**。idx は**自表**文字列で se コード
  （例 `8351A`）。本作 0x6c は 393 件・うち 391 件が `\d{4}[A-Za-z]` の se コードを指す（残り 2 件
  は countdown 台詞で別用途）。
- se コードは `0x01`（汎用コンテンツ表示。書式でテキスト/voice/se/背景を振り分ける）からも参照され、
  **文字列表（=テキスト抽出の NUL 区切りダンプ）に大文字で並ぶ**。`extract_text.py` はこれを
  `[id]` マーカーとして拾う（`8201B`/`9001A` 等）。よって se キューは**テキスト `[id]` マーカー
  （`SE_RE=^\d{4}[A-Za-z]$`）から復元できる**（641 件 / 60 種。0x6c の 393 件より多いのは 0x01 経由分を
  含むため）。se ファイルは小文字（`8351a.ogg`）なので manifest を真実の源に大小文字無視で照合する。
- `_DEF` 文字列表 [78..275] が se シンボル表（`0001a` 〜 `9901c`）。bytecode は se を _DEF index では
  なく**各シーンの自表文字列**で参照する（変数が S 番号直参照なのと同様、_DEF は表示名/定義用）。

→ パイプラインは `parseScene` が `[id]` の se コードを beat に取り込み（`Beat.se`）、`resolveScene` が
manifest で実ファイルへ解決する（`src/pipeline/audio.ts`）。

### BGM = **bytecode に存在しない**（網羅確認済）

M01-M16 の**トラック選択は `md_scr.med` に一切エンコードされていない**。網羅的に確認した:

1. **文字列でない**: `M01`〜`M16` は全エントリの復号後 payload に raw 出現せず。`MUSIC:N` マーカーは
   `_VIEW`（メニュー UI）/ タイトル画面のみで本編シーンに無い。
2. **opcode でない**: 全シーン脚本 ＋ SMAIN の全 (opcode, byte-offset) を走査し、「値が 1-16・出現が
   中程度・1 シーン内少数」という bgm 署名を持つものは皆無（候補は `0x26` の fade 時間上位バイトと
   `0x07` のシーン内連番のみ＝いずれも bgm でない）。`0x2c` は全シーン冒頭に出るが arg は 274/276 が 0。
3. **定義表にもない**: `_DEF`/`_MANPU`/`_MAP`/`MAPCTRL`/`_SYSTEM` に scene→bgm 対応表は無い。

→ 恐らくゲーム実行ファイル側にハードコード。よって BGM は **ルート（character）→ track の
curated 割当**とする（`src/pipeline/audio.ts` の `BGM_BY_CHARACTER`、[ADR 0006](../../../docs/adr/0006-audio-cues.md)）。
エンジンは同 track をシーン跨ぎで継続し、track が変わる遷移でクロスフェードする。

## 3.11 シーン脚本 `[id]` マーカー台帳（HU-40 仕分け）

`build-scenes` は `md_scr_text_<locale>/` のうち **`/^[0-9]/` で始まるシーンファイルのみ** parse する
（`_BGSET`/`_SPRSET`/`_DEF`/`_MANPU`/`EFFECT`/`_VIEW`/`_ALL_dialogue` 等の定義・演出マクロ表は対象外）。
よって `parseScene` の `[id]` 分岐に到達するのは本編シーンのマーカーのみ。各クラスの扱い:

| `[id]` マーカー | 出現 | 扱い |
| -- | -- | -- |
| `<CHAR>_<scene>_<serial>` | 本編 | ボイス（次セリフ beat へ） |
| 4桁+英字（例 `8351A`） | 本編 | se（HU-28） |
| `BG_BLACK` | 本編 | 黒一色背景（HU-35） |
| `ITEM_xx_yy` | 本編 | アイテムCG＝背景（HU-41） |
| `OFF` | 本編 | 立ち絵オフ（HU-36） |
| `BGV_<CHAR>_…` | 本編 | 背景ボイス（ループ・HU-37） |
| `EFFECT:FLASHn` | 本編 | 画面フラッシュ（HU-38） |
| `MIX01/02/03` | 本編 | **無視**＝画像トランジション。CgLayer の既定 1.4s クロスフェードで表現済（HU-38） |
| `_VIEW` ＋後続の小文字 CG コード | 本編 | **無視**＝CG ギャラリー登録メタ（再生表示には無関係。album 機能は将来課題） |
| `_START` | 複合シーン | **無視**＝複合シーン（本文なし→スキップ）の開始印 |
| `SE:` `MUSIC:` `GRA:` `VOL_SET:` `THM_SIZE:` `TYPE:` | 表のみ | **到達せず**＝`_MANPU`/`_DEF`/`EFFECT` 等の演出マクロ表専用（`GRA:BG_RED`＝めまい赤フラッシュ等）。本編シーンには出現しない |
| `PARTS:` `PARTS2:` | `_SPRSET` のみ | **到達せず**＝立ち絵定義の追加パーツ層の区切り。本作は**値が常に空**（パーツ未定義）＝立ち絵は body+face で完結（HU-39。`parseSprset` が skip 済） |

→ 再生反映が必要な未処理マーカーは **無し**（HU-35〜38/41 で出尽くし）。`GRA:` overlay や CG ギャラリー
（`_VIEW`）を将来 UI 機能として実装する場合は、それぞれ overlay 基盤（[FlashLayer] と共通）・album
ビューの新規 issue を立てる。

## 4. 心得・落とし穴

1. **「長さ＝オペコード」と「内側オペコード」を混同しない。** 外側 `len` は単なるデータ長。
   分岐の意味はデータ先頭バイトにある。
2. **`0x08`/`0x09` を物語分岐と誤認しない。** これは 1 シーンの提示ループ（前方スキップ＋後方戻り）。
   実ルート分岐は `0x1c`（hub goto）と各シーン内メニューのフラグ。
3. **hub の「合流」は名前一致、「継続先」はオフセット表。** `SMAIN_MIX02` 等の複数 `0x1c` 参照は同名
   = 同一合流点（in-edge）。一方その先（合流後 goto の継続先＝out-edge）は**ラベル表で解決**する（§3.10。
   HU-22）。文字列表順の単純 zip は終端で誤対応するので、対応は制御フローで確定する。
4. **選択肢テキストは SMAIN に無い。** メニュー文言は各シーン `.bin`/`.txt` 側。
   JP/CN の選択肢 i18n 取り込みは HU-16 の役割。
5. **連番制約でレコード境界を自己検証する。** 長さ表が曖昧でも「次の連番＝今+1」で頑健にパースできる。
   ただし**これは SMAIN 限定**。シーン脚本は `lineno` がソース行＝**非減少（同値の重複可）**で +1
   制約が破綻する（§3.8）。`len` 前置で素直に線形に読むこと（過去 intractable とされた誤同期の罠）。
6. **`08`/`09` の `1e` 区切りで switch の case を数える。** `1d`(len8) switch の各 case は `08 01 <off>`
   ガードで始まり `1e` で区切られる。**case の出現順がフラグ値（1 始まり）**。`08`/`09` の off は
   `data[2:4]` の byte オフセット（`data[1]` の `01` は値ではない・混同注意）。
