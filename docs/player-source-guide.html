まずここだけ

全体像 — ストアが真実、画面は購読者

最初に押さえるのは1文です: Zustand ストア usePlayer が再生状態の単一の真実の源で、物語画面（Stage）とルート図（FlowMap)はどちらも同じストアを購読して追従する。

Stage (PixiJS) 物語の描画と音声。ストアを subscribe し、変化の種類（行送り / beat 送り / シーン遷移）に応じて必要な描画だけ実行
⇄ 購読 / 操作
usePlayer (Zustand) 現在シーン・beat 位置・locale・フラグ・選択肢。シーン跨ぎの遷移は FlowNav（flow.json）に委譲
FlowMap (React Flow) 全ルートの分岐図。再生中ノードをハイライトし、ノードクリックで物語をそのシーンへスキップ
⇄ 同じストアを購読
data/*.json（生成物） シーン・フロー・素材台帳。すべて原データからの機械生成で手編集禁止

アプリの規模感: src/ 全体で約 4,300 行（pipeline 900 / engine+audio 1,300 / flow+store+土台 2,100）、scripts/ が TS 約 1,600 行 + Python 約 1,400 行、electron/ が約 250 行。シーンは jp/cn 各 286 ファイル、フローは 83 ノード / 113 エッジです。

もう一つの柱はデータの出自です。分岐・脚本はすべて原データ（md_scr.med の bytecode）から機械抽出されており、アプリのコードは「生成された JSON をどう再生・描画するか」に専念しています。次節のデータフローを先に頭に入れると、どのコードがどの JSON を読むのかで迷わなくなります。

types.ts — Beat の形を暗記する

Beat は kind で判別する discriminated union です。narration（地の文）は pages（改ページ op 区切りのページ列＝1 ページが 1 送り段）、line（セリフ）は who + lines + voice を持ちます。背景・立ち絵などの見た目は両者共通です。

NarrationBeat { kind:'narration', pages: string[][] , …共通 }
LineBeat      { kind:'line', who, voice?, lines: string[], …共通 }
共通: bg? / sprites?[] / item? / se?[] / lpse? / bgv? / flash?
// 素材参照はすべて file: string | null — null は「未解決」を表し validate が検出する

入力側の SceneEvent は ['text', …] ['bg', …] ['sprite', …] のようなタグ付きタプルで、bytecode の opcode をほぼ 1:1 で写したものです。

scene.ts — buildScene の状態機械

buildScene (scene.ts:65) は 2 種類の状態を持って走査します:

  • sticky（次に上書きされるまで持続）: 背景・被せ CG・立ち絵スロット・アイテム窓・BGV・ループ se。beat 確定のたびに snapshot() で焼き込むので、「背景や BGV がずっと続く」が実現される
  • pending（使い切り）: 話者・ボイス・se・フラッシュ。次の beat に一度だけ載る

設計上の要点は 2 つ。①「見た目署名（visSig）」が変化したときだけ地の文 beat を区切る＝立ち絵の内部操作で文章が分断されない。②話者決定は pendingSpeaker 最優先 → ボイス ID 接頭辞の学習辞書 → 直前話者 → 主人公、の順のフォールバック。冒頭の JSDoc（1〜31 行）にこの全体設計がまとまっているので、本体より先に読んでください。

resolve.ts / defs.ts / audio.ts — 解決の三兄弟

  • voice の大小文字の罠: ボイス ID AYAN_002_AYAN001A_001 の実ファイルは ayan_002_ayan001A_001.ogg（末尾の変種文字だけ大文字）。単純な小文字化では一致しないため、manifest を真実の源に小文字化キーで照合します (resolve.ts:19)
  • 立ち絵/背景は [note] ラベルを sprites.json/backgrounds.json と最長一致 prefix で照合し body+face コードへ解決 (defs.ts)
  • BGM は原データに存在しない（トラック選択は実行ファイル側と推定）。そのためルート（character）→ M01–M16 の BGM_BY_CHARACTER による curated 割当で、対応表は編集可能です (audio.ts:52)

Stage の購読 — 変化の粒度で副作用を絞る

usePlayer.subscribe (Stage.tsx:177) が新旧 state を diff し、必要な処理だけ呼び分けます。これがエンジン理解の最重要ポイントです:

  • line だけ変化（行送り）→ renderText のみ。背景・立ち絵・音声は据え置き
  • index 変化（beat 送り）→ renderBeat（全描画＋音声発火）
  • scene 変化 → シーン遷移なら onSceneChange（BGM 切替・旧テクスチャ/ボイスの遅延解放）＋ renderBeat。locale だけの変化（言語切替）なら silent 付きで字幕だけ描き直し、音声もテクスチャも乱さない

全レイヤに共通するパターンは「同一入力なら no-op・変化時のみフェード・ロード中に陳腐化したらキャンセル」。tween() が返すキャンセル関数を各レイヤが保持し、連打で beat が進んでも破綻しません。

src/audio/ — 209 行

audio — 5 チャンネルの寿命を管理する

AudioManager は Howler の薄いラッパですが、チャンネルごとに寿命が違うのが肝です。

チャンネル性質寿命切替
voice台詞。前を止めて 1 本再生beat 同期即切替（URL キャッシュ）
seワンショット・多重再生可単発—
bgmループ・音量 0.4（curated）シーン跨ぎ継続track 変化時のみ 800ms クロスフェード
bgv背景ボイス・単一ループ・stickyシーン局所即切替
lpseループ se（HU-76、bgv と同型）シーン局所即切替

Howler 運用の要点は 2 つ。①キーボード操作でも音を出せるよう、再生前に必ず suspended な AudioContext を resume() する（withContext）。②生成直後の play() は無音になり得るため、state()==='loaded' を確認し未ロードなら once('load') で待ってから再生。シーン離脱時は releaseVoices() で voice キャッシュを unload し、メモリを抱えません。

src/store/ — 337 行

store — 再生位置とブックマーク

アプリの中枢。player.ts（208 行）が再生状態、bookmarks.ts（129 行）がセーブデータを持ちます。

// PlayerState（player.ts:32）の形
scene: Scene | null      // 現在シーン（beat 配列を含む）
index: number            // 現在の beat
line:  number            // beat 内サブ位置（narration のページ送り）
locale: 'jp' | 'cn'
flags: ReadonlySet<string>      // ルート分岐フラグ（シーン跨ぎ保持）

advance() (player.ts:106) の判定は 4 段: ①選択肢提示中は無視 → ② beat 内に次ページがあれば line+1 → ③シーン内に次 beat があれば index+1 → ④シーン末尾なら FlowNav.advance() に委譲し、結果が scene（次シーンをロード）/ choice（選択肢提示）/ end（終端）のいずれかになります。

FlowNav (src/flow/nav.ts) は flow.json を辿る純ロジックで、単一出エッジは自動進行、複数エッジは選択肢化、空ノード（start/hub）は透過的に解決します。setLocale() は現在シーンを別言語で読み直し、再生位置を維持します（ベストエフォート）。

bookmarks.ts は 1 シーンにつき 1 件を上書き保存。Electron では preload が公開する saimin4Desktop.bookmarks 経由でファイル（userData/bookmarks.json）へ、Web では localStorage へフォールバックします。起動時は最新ブックマークへ自動復帰します（start()）。

クリックから次の表示まで

「進める」操作 1 回で何がどの順に起きるか。この 1 本を追えると、ストア・エンジン・音声の役割分担が体で分かります。

  1. 入力 — クリック / Space / Enter / → を Stage が受け、usePlayer.getState().advance() を呼ぶ (Stage.tsx:216)
  2. 判定 — advance() が 4 段判定: 選択肢中は無視 / beat 内の次ページ（line+1）/ 次 beat（index+1）/ シーン末尾なら FlowNav.advance() → 次シーンロード or 選択肢提示 or 終端 (player.ts:106)
  3. 購読が diff — Stage の subscribe が変化の種類を見て、行送りなら renderText のみ、beat 送りなら renderBeat、シーン遷移なら onSceneChange（BGM 切替・旧素材の遅延解放）＋ renderBeat (Stage.tsx:177)
  4. renderBeat の実行順 — 背景クロスフェード → 立ち絵 → アイテム窓 → 字幕/題字 → voice → se → bgv → lpse → flash。言語切替時は silent で音声・テクスチャに触れない (Stage.tsx:113)
  5. 選択肢の場合 — pendingChoice がセットされると React の ChoiceOverlay がボタンを描画し、クリックで choose(target) → 分岐先シーンをロードして 3 に戻る

やってはいけない・ハマりどころ

規約と落とし穴

  • data/*.json は手編集禁止 — flow.json・scenes・sprites などすべて生成物。再生成で上書きされます
  • ../prototype/ を改変しない — 演出パラメータ・仕様の参照源（picturebook_v3.html / route_map.html）
  • CG・音声・original_game/ をコミットしない — リポジトリは public。素材は git 外で manifest のみ管理（Git LFS も不使用）
  • ボイス ID の大小文字は推測しない — 変換規則が不規則なので manifest 照合が唯一の正解
  • beat 生成は bytecode 一次 — 人間可読の txt（md_scr_text_*）は文字列デデュープで再表示が欠落するため beat には使わない（参照・validate 用に残置）
  • git は GitHub Flow — main は保護ブランチ（PR + CI 必須）。ブランチは hzq886/hu-<N>-<slug>（Linear issue 単位）。PR 前にローカルで typecheck / lint / format:check / test を緑に
  • スキーマ・解決規則・素材取得方式・フロー復元方式を変えるときは ADR を書く（docs/adr/）
